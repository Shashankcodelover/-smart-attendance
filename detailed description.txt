## 1. Why It Is Being Created (The Problem Statement)

In modern university environments, traditional roll-calls and manual attendance logs create massive operational overhead and security loopholes:

* **Teaching Time Erosion:** In a typical 50 to 60-minute lecture, manually calling out roll numbers or circulating paper logs consumes between 10 to 15 minutes of class time. This represents a direct 20% to 25% loss in active instructional velocity.
* **Network Connectivity Failure:** Existing digital attendance systems rely heavily on stable, continuous internet connectivity. In lecture halls built with reinforced concrete, cellular signals and campus Wi-Fi are notoriously unreliable. When a student's network goes down, the digital app completely fails, forcing a fallback to manual signing and rendering digital records inaccurate.
* **The "Proxy" Threat:** Standard digital attendance links or geo-location check-ins can easily be bypassed. Students share direct links or GPS coordinate spoofers via messaging apps, allowing peers physically sitting in dorm rooms to register attendance ("Proxies"), which undermines grading integrity and record auditability.
* **Biometric & Intrusive Privacy Audits:** Fingerprint scans or facial recognition solutions raise heavy privacy concerns, require expensive, specialized proprietary hardware, and face severe compliance roadblocks internationally.

---

## 2. What It Is Going to Solve (The Unique Solution)

The Smart Offline Web-Based Attendance System solves classroom administrative overhead and proxy inflation by introducing a **High-Velocity, Cryptographic QR-OTP Session Gateway** running on an offline-first architecture:

1. **Session-Driven Control Plane:** Attendance is managed strictly by the lecturer's direct trigger. Active session codes are only generated on-demand during class.
2. **Ephemeral Dynamic QR Generation:** Instead of static codes, the system generates a dynamic QR containing a signed cryptographic token valid only for a short, lecturer-controlled duration (e.g., 30 seconds).
3. **Double-Factor OTP Verification:** After scanning the QR code, the student is prompted to enter a real-time 4-digit PIN generated on the lecturer's screen. This dual-verification (scanning the physical screen + entering the immediate visual code) ensures the student is physically present in the room.
4. **Resilient Offline Buffering:** If a student's mobile device loses internet connection during the scan, the client-side engine intercepts the failure, serializes the cryptographically verified receipt into the browser's `localStorage` queue, and immediately displays a "Saved Offline" visual confirmation.
5. **Autonomic Network Reconciliation:** A background sync daemon continuously checks network health. The moment internet connectivity is restored on the student's browser, the queued payloads are automatically replayed and synced to the database.

## 4. Fully Detailed Tech Stack

To ensure that this platform can run immediately on standard local servers or single-board computers in server-deprived locations, we use a lightweight, high-speed stack:

* **Frontend Client (Vanilla Web):** Mobile-first HTML5, CSS3, and JavaScript utilizing the native `Html5Qrcode` scanner API. The UI relies on HSL variables for an interactive, modern visual layout.
* **Application Server (Node.js / Express 5.2.1):** Leverages Express v5 for asynchronous route handlers and high-performance pipeline routing.
* **Local Database Engine (SQLite + `better-sqlite3`):** Rejects heavy network database setups in favor of a fast, local transactional C-compiled SQLite engine, guaranteeing atomic, lightning-fast queries:

```sql
CREATE TABLE IF NOT EXISTS sessions (
  id TEXT PRIMARY KEY,
  subject_code TEXT NOT NULL,
  otp TEXT NOT NULL,
  status TEXT DEFAULT 'INACTIVE',
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  expires_at TIMESTAMP
);

CREATE TABLE IF NOT EXISTS attendance_records (
  id TEXT PRIMARY KEY,
  session_id TEXT NOT NULL,
  student_usn TEXT NOT NULL,
  marked_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  marked_online INTEGER DEFAULT 1,
  FOREIGN KEY(session_id) REFERENCES sessions(id)
);
```

* **Dynamic Networking (WebSockets via `ws` v8.21.0):** Drives real-time synchronization between the student scans and the lecturer dashboard, allowing the lecturer to watch their room's completion bar populate dynamically.
* **Geospatial Distance Estimations:** Implements offline-friendly validations, validating local timestamps to enforce:

$$\Delta t = t_{\text{client}} - t_{\text{server}} \le 30 \text{ seconds}$$

This restricts submission delays and guarantees that offline-logged entries were captured precisely inside the active session timeframe.

## 5. Phase-by-Phase Workflow Functional Description

### Phase 1: Lecturer Session Generation

1. The lecturer signs in to the portal and selects the active subject code (e.g., `01JST24UCS100 — Computer Networks`).
2. The server instantiates a session record in SQLite, creating an active token and a random 4-digit numeric code.
3. The server generates a signed QR payload containing the session token:

$$\text{QR Payload} = \{ \text{token: "JST24-99823", t: } 1774883920 \}$$

### Phase 2: QR Scanning and OTP Handshake

1. The student navigates to the portal on their smartphone, types their validated USN, and taps "Open Scanner".
2. The camera is activated using standard browser camera controls, scanning the visual QR displayed on the lecturer's projector screen.
3. Once scanned, the student is presented with a 4-digit OTP numeric entry grid.
4. The student reads the live OTP PIN shown on the projection screen and enters it.
5. Tapping "Submit" triggers the authentication payload to the server.

### Phase 3: Offline Buffering & Local Receipt Creation

1. If the student's phone has active cellular data, the POST request is processed, and a "Marked Online" confirmation is displayed.
2. *The Offline Case:* If the browser detects a network failure (timeout or standard browser offline event), the client-side JavaScript intercepts the error:
   * It compiles the submission details into an offline JSON payload.
   * It saves this string under `localStorage.setItem("offline_attendance_payload", JSON.stringify(payload))`.
   * It immediately changes the interface to a "Saved Offline" receipt screen showing an amber badge.

### Phase 4: Autonomic Background Reconciliation

1. The student's browser continuously listens for the standard network event: `window.addEventListener('online', ...)`.
2. When the phone connects to a cellular tower or standard Wi-Fi, the online listener triggers the background sync function.
3. The background sync function fetches the buffered payload from `localStorage`, sends it to the server API, and removes the payload upon a successful 200 HTTP response.
4. The user is notified with a celebratory non-intrusive toast: *"Offline attendance successfully synchronized! 🎉"*.
